# battle-domain 设计说明

## 目标

`battle-domain` 用纯 Rust 值表达可复现的双人对战。
同一队伍、命令序列和种子必须产生相同的对战状态和事件顺序。
领域代码不读取外部资源，也不产生外部副作用。

## 模块划分

`model.rs` 定义队伍、宝可梦、招式、能力值、状态和输入校验。
`stats.rs` 按第三世代整数公式将基础种族值、等级和训练参数投影为战斗能力值。
`rules.rs` 定义属性相性、伤害类别和可复用的伤害修正规则。
`battle.rs` 持有对战状态，校验命令并结算回合。

这种划分让值对象和静态规则不依赖状态机。
`Battle` 只协调已有模型和规则，不向调用方暴露内部模块布局。

## 聚合与不变量

`Battle` 是两支队伍、出战槽位、待处理命令、伪随机状态、回合编号、事件历史和天气的唯一容器。
构造时，两支队伍都必须有存活成员，且跨队伍的 `PokemonId` 必须唯一。

`Team` 恰好包含六只标识不同的宝可梦。
`Pokemon` 的等级在 1 至 100 之间，当前 HP 不超过最大 HP，且携带一至四个标识不同的招式。
`Move` 的最大 PP 大于零，当前 PP 不超过最大 PP。
这些限制由构造函数返回 `ValidationError` 表达。

## 命令与事件

`Battle::legal_actions` 是合法命令的唯一查询入口。
它根据当前阶段、已提交命令、PP、存活成员和限制换人的特性生成动作。
`Battle::submit` 先在候选状态上执行校验和结算，成功后才替换原状态，因此错误不会留下部分状态或事件。

事件是按结算顺序追加的事实记录。
`SubmitOutcome` 只返回本次提交产生的事件，方便调用方增量渲染或持久化。
完整历史由 `Battle::events` 持有。

## 演进约束

新增静态属性相性或数值规则时，应放入 `rules.rs` 或 `stats.rs`，并保持函数无副作用。
新增会改变回合、队伍或事件的规则时，应放入 `battle.rs` 并为成功与非法命令路径添加测试。
新增可见 API 时，必须从 `lib.rs` 明确导出，避免调用方依赖私有模块路径。
