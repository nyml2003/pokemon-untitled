# Lint 与绑定规则

## Clippy

- 运行 Clippy 后逐条解决根因。优先调整 API、数据流、所有权、分支或模块边界。
- 不新增 `#[allow(clippy::...)]`、`#[expect(clippy::...)]`、crate 级 Clippy 压制，或通过命令行降低 lint 级别。
- 不以重命名、死代码、空分支、无意义克隆或更复杂的控制流消除警告。删除无用代码，或让代码承担明确职责。
- 既有压制不是新增压制的理由。修改含压制的函数时，优先重构以移除它；确实超出当前任务时，不要扩大其范围。

## 未使用绑定

- 值、局部变量、导入、泛型参数和函数参数没有用途时，先删除。
- 接口、trait 实现、回调或模式匹配要求保留位置时，使用精确的 `_`。
- 不使用 `_name`、`_unused`、`_ignored` 等带名字的未使用绑定。它们会隐藏应删除或应重构的设计问题。
- 需要仅为了触发副作用而调用函数时写 `let _ = call();`；只有明确丢弃该返回值且副作用是所需行为时才允许。
- 不为保留无用绑定添加 `#[allow(unused_variables)]`、`#[allow(dead_code)]` 或同类压制。

## 改造顺序

1. 删除不参与行为的变量、参数、导入和中间结果。
2. 删除参数后同步所有调用方、trait 定义和测试。
3. 若接口由外部契约固定，将未使用参数改为 `_`。
4. 若值本应参与业务行为，补齐该行为或拆分函数，不保留“未来可能会用”的变量。

## 验证

使用项目的 ops 入口验证：

```text
ops format --check
ops test
```

范围很小且项目已提供对应 suite 时，可使用适配的 `ops test --suite ...`。报告没有运行的验证及原因。
