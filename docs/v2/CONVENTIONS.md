# 文档维护规则

> 分类：现状；最后核对：2026-07-20

## 分类规则

每篇 `docs/v2` 文档开头必须明确写出分类。

- `现状`：每个关键结论都能指向当前代码、配置或验收结果。写明“最后核对”日期。
- `未来`：描述目标、边界、前置条件和完成标准。不得使用“已实现”描述尚未验收的内容。
- `过时`：不建立目录，不做副本，不保留跳转页。删除文件；历史由 Git 保存。

没有足够证据证明为现状的内容，一律归入未来，或先删除。

## 存放位置

| 内容 | 位置 |
| --- | --- |
| 跨 crate 的已实现架构、运行流程和开发方式 | `docs/v2/current/<编号-主题>/<编号-内容>.md`；主题 `README.md` 只做导航。 |
| 跨 crate 的提案、产品方向和待决策事项 | `docs/v2/future/` |
| 单个 crate 的 API、状态不变量和设计取舍 | 该 crate 的 `README.md` 或 `docs/design.md` |
| 资产来源、许可证和导入说明 | 资产目录内 |
| 一次性实施记录、已失效分析和旧方案 | 删除 |

## 更新时机

出现下列变化时，在同一变更中更新对应文档。

1. 修改 workspace 成员、依赖方向、运行时入口或权威状态所有者时，更新受影响主题的 `README.md` 导航和对应编号内容页。
2. 修改 UI 路径、页面迁移范围或渲染边界时，更新 `current/007-UI/` 下对应编号内容页。
3. 修改开发、同步、构建或目标平台验收方式时，更新 `current/008-开发与验收/` 下对应编号内容页。
4. 未来条目开始实现时，先把已完成部分移入 `current/`；未完成部分保留在 `future/`。
5. 某项方案被否决、替代或完成后，删除旧条目。不要把完成日志留在未来文档中。

## 写作要求

- 一篇文档只表达一种分类。不要混入“当前实现”和“下一阶段计划”。
- 现状文档写事实、边界和证据位置，不写实现愿望。
- 未来文档写决策输入和验收条件，不复制当前源码细节。
- 不用日期命名常驻文档；日期只用于“最后核对”或一次性发布记录。

## 图示与校验

架构图是正文证据的辅助表达，不替代文字说明、代码位置或验收结论。仅在关系、时序或状态转移无法由短文本清楚表达时使用。

- Mermaid 用于分层、数据流、状态机和较短的运行流程；PlantUML 用于参与者较多的时序图或组件关系图。
- 一张图对应一个明确问题，并放在说明该问题的编号内容页中。正文必须在图前说明范围、在图后解释结论和代码证据。
- 图示源码与所在页面同目录：Mermaid 使用同名 `.mmd`，PlantUML 使用同名 `.puml`；二者都提交同名 `.svg` 渲染产物。页面引用 SVG，不直接依赖宿主对 Markdown 图示语法的支持。
- 图示文件名沿用页面编号，例如 `004-场景投影与帧提交-运行流程.mmd`、`004-场景投影与帧提交-运行流程.svg`。一个源码文件只生成一个 SVG。
- 新增或修改图示前，安装并使用持久化 Nix profile 中的 `mmdc`、`plantuml` 和 `dot`。工具安装命令为 `nix profile add nixpkgs#mermaid-cli nixpkgs#plantuml nixpkgs#graphviz`。
- 校验时必须将源码渲染到仓库外的临时目录，再与提交的 SVG 比对；Mermaid 使用 `mmdc -i <源码.mmd> -o <临时.svg>`，PlantUML 使用 `plantuml -tsvg -o <临时目录> <源码.puml>`。渲染失败或产物不一致时不得提交。
- SVG 比对通过后，仍须打开渲染结果作视觉复核，确认中文字、箭头方向、布局和边界均可读。图示变更说明中记录渲染器版本和校验结果。
