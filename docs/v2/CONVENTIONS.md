# 文档维护规则

> 分类：现状；最后核对：2026-07-20

## 分类规则

每篇 `docs/v2` 文档开头必须明确写出分类。

- `现状`：每个关键结论都能指向当前代码、配置或验收结果。写明“最后核对”日期。
- `未来`：描述目标、边界、前置条件和完成标准。不得使用“已实现”描述尚未验收的内容。
- `过时`：不建立目录，不做副本，不保留跳转页。删除文件；历史由 Git 保存。

没有足够证据证明为现状的内容，一律归入未来，或先删除。

## 存放位置

| 内容 | 位置 |
| --- | --- |
| 跨 crate 的已实现架构、运行流程和开发方式 | `docs/v2/current/` |
| 跨 crate 的提案、产品方向和待决策事项 | `docs/v2/future/` |
| 单个 crate 的 API、状态不变量和设计取舍 | 该 crate 的 `README.md` 或 `docs/design.md` |
| 资产来源、许可证和导入说明 | 资产目录内 |
| 一次性实施记录、已失效分析和旧方案 | 删除 |

## 更新时机

出现下列变化时，在同一变更中更新对应文档。

1. 修改 workspace 成员、依赖方向、运行时入口或权威状态所有者时，更新 `current/architecture.md`。
2. 修改 UI 路径、页面迁移范围或渲染边界时，更新 `current/ui.md`。
3. 修改开发、同步、构建或目标平台验收方式时，更新 `current/development.md`。
4. 未来条目开始实现时，先把已完成部分移入 `current/`；未完成部分保留在 `future/`。
5. 某项方案被否决、替代或完成后，删除旧条目。不要把完成日志留在未来文档中。

## 写作要求

- 一篇文档只表达一种分类。不要混入“当前实现”和“下一阶段计划”。
- 现状文档写事实、边界和证据位置，不写实现愿望。
- 未来文档写决策输入和验收条件，不复制当前源码细节。
- 不用日期命名常驻文档；日期只用于“最后核对”或一次性发布记录。
