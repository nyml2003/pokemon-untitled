# 参与者与入口

> 分类：现状；最后核对：2026-07-20。
> 依据：各入口 package 的 `Cargo.toml` 与 `src/main.rs`。

## 参与者

同一 workspace 服务于玩家、内容创作者和开发自动化，但他们不通过同一个入口工作。

| 参与者 | 入口 | 输入 | 结果和副作用 |
| --- | --- | --- | --- |
| 玩家 | `game-host` | 键盘、文字、鼠标、内嵌游戏数据、地图、资产和叙事脚本。 | 更新 `GameSession`，向原生窗口提交帧。 |
| 地图作者 | `map-editor` | 可选项目路径、瓦片资产和语义定义。 | 编辑 `MapProject`，保存 JSON 或 `.g3mp` 项目。 |
| 自动化或地图作者 | `map-editor-cli` | 子命令和地图文件或目录。 | 输出检查或 lint 结果，读写打包、解包后的文件。 |
| 内容作者 | `tile-editor` | 资产目录和瓦片语义数据。 | 在原生窗口编辑瓦片语义。 |
| 数据维护者 | `game-data-import` | CSV 源目录、目标文件、版本组和可选 locale。 | 生成并重新校验游戏数据 JSON。 |

## 入口的输入归一化

`game-host` 不把 Winit 原始事件直接交给游戏规则。它通过 `WinitKeyEventSnapshot`、`normalize_key_event` 和 `normalize_committed_text` 把键盘和文字事件转为项目输入，再交给 `PresentationState` 或 `GameSession`。窗口尺寸变化只调整 native target 和重绘请求。

`map-editor` 与 `tile-editor` 同样在 runtime 中接收 Winit 事件，但编辑意图和编辑状态位于 `map-editor-core`、`tile-editor-core` 等核心 crate。`map-editor-cli` 没有窗口循环；它先解析子命令和路径，再调用地图模型、语义和存储 API。

## 进程级错误

入口负责把失败变为用户可见的进程结果。`game-data-import` 解析参数或生成失败时向标准错误输出并返回失败退出码。`map-editor-cli` 对非法参数、缺失输入和校验失败返回错误。原生窗口程序在启动、读取、投影或提交失败时由 runtime 终止或退出事件循环。

这些结果不等于领域错误模型：`MapStorageError`、`GameError` 等结构化错误先由下层返回，入口决定如何显示、记录或映射为退出状态。
