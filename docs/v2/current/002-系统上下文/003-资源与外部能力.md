# 资源与外部能力

> 分类：现状；最后核对：2026-07-20。
> 依据：资源目录、入口源码与 `game-fs-assets`、`map-project-storage`、`game-native-target`、`punctum-wgpu`、`punctum-crossterm`、`battle-ramus-adapter`。

## workspace 资源

`assets/`、`maps/` 和 `narrative/` 位于 workspace 根目录，供多个入口共同使用，不属于某一个 crate。`game-host`、`map-editor`、`map-editor-cli` 和 `tile-editor` 都从各自 `CARGO_MANIFEST_DIR` 向上解析这些目录。

| 资源 | 当前形式 | 使用边界 |
| --- | --- | --- |
| `assets/` | 目录、目录清单、图像和生成的数据文件。 | runtime 选择路径；`game-fs-assets` 读取内容；presentation 根据资产计划描述需要的资源。 |
| `maps/` | `.json` 地图与 `.g3mp` 容器。 | 地图模型负责合法性；runtime/CLI 选择文件；adapter 处理二进制容器。 |
| `narrative/` | `.narrative` 脚本。 | `game-host` 目前用 `include_str!` 编译期嵌入 demo 脚本。 |
| `game-data` | 内嵌只读数据。 | `game-data` 用 `include_bytes!` 读取已生成的数据文件并构造 `CurrentDataSet`。 |

路径和目录扫描属于 runtime/adapter，不能进入游戏规则或视图模型。

## adapter 边界

| 外部能力 | 实现 | 面向内层的结果 |
| --- | --- | --- |
| 本地资产读取 | `game-fs-assets` | `AssetCatalog`、`TileSource`、`AssetBytes` 或 `io::Error`。 |
| 地图二进制容器 | `map-project-storage` | 已校验的 `MapProject`、元数据、字节或 `MapStorageError`。 |
| 原生窗口、输入和 GPU | `punctum-wgpu` | 已归一化输入、`GpuRuntime`、`PresentOutcome` 与 `GpuRuntimeError`。 |
| 原生帧与文本提交 | `game-native-target` | `FramePlan` 到 native GPU 提交的 `NativeTarget`。 |
| 终端 | `punctum-crossterm` | 终端输入归一化、session 和 presenter。 |
| 对战命令能力 | `battle-ramus-adapter` | 经过授权、解析与能力校验的战斗 `Action`。 |

例如，`game-fs-assets::read_asset_requests` 按 `AssetRequest` 在目录清单中定位资源，再返回字节；它不会把 `PathBuf` 交给 application。`BattleRamusAdapter` 只公开目录中已注册且授权的移动、换人和挣扎指令，未知路径、额外参数或不匹配授权会返回 adapter diagnostic。

## 存储和提交的失败模型

`map-project-storage` 的 `.g3mp` 容器有固定 magic、版本、大小上限、Zstd 压缩和 payload 校验和。读取或写入会把截断、版本不支持、校验和不符、压缩失败、尺寸超限和 `MapProject` 校验失败表示为 `MapStorageError`。

`punctum-wgpu` 把 surface 的正常跳过、重配与丢失表示为 `PresentOutcome`，把不可恢复问题表示为 `GpuRuntimeError`。`game-native-target` 再将 GPU 与文本编码错误收束为 `NativeTargetError`。这些类型留在 adapter/runtime 边界；内层只消费投影输入、帧计划或业务结果。
