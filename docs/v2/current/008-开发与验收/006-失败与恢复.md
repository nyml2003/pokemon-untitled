# 失败与恢复

> 分类：现状；最后核对：2026-07-20。
> 依据：`ops-first-workflow` 的失败分类、阶段输出与恢复约束。

## 保留可诊断结果

长任务将阶段进度和子进程输出写入标准错误；使用 `--json` 时，标准错误是一行一个 JSON 事件，标准输出只包含最终 JSON 结果。失败时先保留这些记录，再判断是环境、镜像、LFS、同步、构建还是运行阶段的问题。

| 结果 | 恢复方式 |
| --- | --- |
| `MirrorMissing` | 执行 `ops init-mirror`。 |
| `MirrorDirty`、`MirrorDiverged` | 停止，人工确认镜像内容；不自动修改。 |
| `GitLfsUnavailable` | 修复 Git LFS 环境后重试需要 LFS 的阶段。 |
| `GitSyncFailed` | 修复远端、网络或提交状态后重试同步。 |
| `BuildFailed`、`RunFailed` | 保留错误事件，修复对应 native 环境或提交后重试。 |
| `Cancelled` | 不假定镜像已到目标提交；先运行 `ops check` 再重启所需操作。 |

## LFS 与重试边界

Git LFS 只在镜像初始化、`.gitattributes` 改变或新增/修改 LFS 指针时运行。普通源码提交和已对齐的镜像会明确报告跳过 LFS。不要因为一次普通同步而强制执行额外 LFS 操作。

恢复时不得手工创建、覆盖或清理镜像目录，也不得自动 stash、clean、merge、rebase 或 reset。恢复的目标是让 `ops check` 重新确认一个明确状态，而不是用隐式破坏性操作掩盖失败。
