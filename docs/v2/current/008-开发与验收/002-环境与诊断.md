# 环境与诊断

> 分类：现状；最后核对：2026-07-20。
> 依据：`flake.nix`、`ops.local.example.json`、`ops-first-workflow` 与 `AGENT.md`。

## 开发环境

在 workspace 根目录进入 `nix develop` 后执行 `ops`。Nix 开发环境提供项目约定的工具和入口；日常工作不以宿主机随机安装的构建、同步或 Windows 命令替代它。

首次使用或环境变化后，先检查配置和运行端：

| 命令 | 用途 |
| --- | --- |
| `ops check` | 检查配置、镜像和运行端是否满足工作流前提。 |
| `ops doctor` | 输出更详细的环境诊断。 |
| `ops check --json` 或其他 `--json` 形式 | 获取机器可读的最终结果和标准错误阶段事件。 |

`ops.local.json` 是本地配置文件；以 `ops.local.example.json` 的字段结构配置实际环境。它缺失或 `ops doctor` 失败时，应先修复配置或 ops 本身，不退回到其他构建和同步路径。

## 新 Windows 镜像

新的 Windows Git 镜像必须先由 `ops init-mirror` 显式初始化。初始化目标只能是空目录；`ops sync`、`ops build game-host` 和 `ops run game-host` 不会创建、覆盖或清理镜像目录。

初始化后先运行 `ops check` 确认状态，再开始同步或原生验收。镜像是配置远端分支提交的工作副本，不是编辑源，也不是 WSL 测试的替代环境。
