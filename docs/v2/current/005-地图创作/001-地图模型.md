# 地图创作

> 分类：现状；最后核对：2026-07-20
> 依据：`map-project`、`map-tile-semantics`、`map-editor-core`、`map-editor-view`、`map-project-storage`、`map-editor` 和 `map-editor-cli`。

## 模型与状态所有权

| 内容 | 所有者 | 作用 |
| --- | --- | --- |
| 地图项目、地图 ID、图块、碰撞、事件、角色和编辑历史 | `map-project` | 定义可校验、可序列化、可逆编辑的地图文档。 |
| 瓦片语义目录与诊断 | `map-tile-semantics` | 校验分层图块规则，并返回地图语义诊断。 |
| 编辑工具、选择、历史、视图范围和编辑效果 | `map-editor-core` | 将 `EditorIntent` 归约为下一个 `EditorModel`。 |
| 瓦片语义编辑状态 | `tile-editor-core` | 将瓦片语义 action 归约为新目录状态。 |

地图编辑规则在 `map-project` 和 `map-editor-core`。视图、文件路径、窗口和磁盘写入不属于这些 crate。

## 原生地图编辑器

```text
Winit 输入
  -> map-editor-core 的输入和 controller
  -> EditorIntent
  -> EditorModel reducer
  -> map-editor-view::project
  -> EditorFrame
  -> FramePlan
  -> NativeTarget::present
```

`map-editor-view` 只读取 `EditorModel` 和原子图块目录。它生成工作台界面，并把命中的 UI action 映射为 `EditorIntent`。runtime 决定何时调用 reducer、何时保存项目和怎样展示错误。

## 命令行地图工具

`map-editor-cli` 支持交互式虚拟命令，以及 `inspect`、`verify`、`pack`、`unpack` 和 `lint` 子命令。

交互式路径将 JSON `EditorVirtualCommand` 交给 `map-editor-core`。子命令在 runtime 边界读取或写出文件，并使用 `map-assets`、`map-project-storage` 和 `map-tile-semantics` 构造或校验数据。

## 地图存储

`map-project` 提供 JSON 表示和地图有效性检查。`map-project-storage` 负责 `.g3mp` 二进制容器：

- 容器魔数为 `G3MP`，当前容器版本为 1。
- 写入前校验地图、尺寸和载荷限制。
- 写入时使用 zstd 压缩并记录 payload checksum。
- 读取时检查头部、版本、长度、压缩数据、checksum 和解码后的地图。
- `MapProjectReader` 和 `MapProjectWriter` 只接受字节、`Read` 或 `Write`。它们不决定文件路径。

具体 `std::fs` 调用留在 `map-editor` 或 `map-editor-cli`。因此同一个存储 crate 可用于内存测试和其他文件来源。

## 游戏加载

`game-host` 读取地图项目和瓦片资源后，使用 `WorldApplication::from_map_project_with_scripts` 构造世界用例状态。地图碰撞和遭遇事件被转换为世界地表；角色、外观和叙事脚本在 application 边界绑定。
