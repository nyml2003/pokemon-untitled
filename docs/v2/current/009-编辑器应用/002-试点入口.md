# 试点入口

> 分类：现状；最后核对：2026-07-20。
> 依据：`map-editor-core`、`trainer-editor-core`、`pokemon-editor-core`、`trainer-editor-cli` 与 `pokemon-editor-cli`。

| 内容 | 纯 core | 统一协议接入 | 当前入口 |
| --- | --- | --- | --- |
| 地图 | `EditorModel` 与 `EditorVirtualCommand` | `virtual_command_from_editor_call` 将 map `EditorCall` 解码为既有虚拟命令。 | `map-editor`、`map-editor-cli`。 |
| 训练师 | `TrainerEditorModel` 与 `TrainerEditCommand` | 模型校验姓名、队伍和脚本；GUI 与 CLI 都将命令经 Ramus 后执行。 | `trainer-editor` Windows GUI、`trainer-editor-cli` JSON Lines。 |
| 宝可梦 | `PokemonEditorModel` 与 `PokemonEditCommand` | 模型校验 ID、图鉴号、名称、属性与基础 HP；GUI 与 CLI 都经 Ramus 执行命令。 | `pokemon-editor` Windows GUI、`pokemon-editor-cli` JSON Lines。 |

训练师和宝可梦 CLI 从各自固定的逻辑 document ID 装载 catalog。每一行输入是 `EditorCall` JSON；入口先调用 `EditorRamusRouter::route_call`，只在得到已授权的路由意图后才调用 core 或资源 adapter。`Save` 通过注册表写回，成功后才清除 dirty 状态。

`trainer-editor` 是 Windows 原生 GUI。窗口使用 Punctum UI 显示姓名、脚本、队伍、状态和保存动作；姓名和脚本通过 IME 输入，队伍可添加或移除最后一只成员。每次 UI 修改先构造 `EditorCall`，再由 Ramus 路由到 `TrainerEditorModel`；保存同样经过路由并由资源 adapter 写回。

`pokemon-editor` 使用同一套 Winit/Punctum UI 运行时，支持名称 IME 输入、属性切换、基础 HP 步进和保存。每个修改先构造 `EditorCall`，保存通过受控逻辑资源写回 catalog。

Windows 原生调度的公开目标现在包括 `game-host`、`map-editor`、`trainer-editor` 和 `pokemon-editor`。调度器按目标选择 Windows 私有 runner 中的 binary 名称，仍先同步镜像。当前已验证的编辑器自动化入口是两个 JSON Lines CLI、两个 GUI 的 workspace 编译与既有地图工具。Windows GUI 的实际窗口验收需要先同步本次变更到镜像。
