# 迁移边界

> 分类：现状；最后核对：2026-07-20。
> 依据：`punctum-ui` 的弃用 API、当前场景投影和 `docs/v2/future/README.md`。

## 当前已验收的能力

当前 UI 有两条可用路径：地图/编辑器的 Grid 路径，以及游戏页面和覆盖层的 Pixel UI 路径。两者都能投影为 `FramePlan` 并由 native runtime 提交。Pixel UI 已具备 tree 构建、结构 ID、动态 key、布局、裁剪、绘制命令、命中区域和类型化 action。

这些是现状文档可以承诺的范围。页面需要的资源、按钮或视觉结构应在这两个既有模型内表达。

## legacy 的位置

`UiNode::new`、`UiNode::legacy` 与 `UiFrame::hit_test` 仍保留在 API 中，但都标记为 deprecated。已有调用可以在不改变行为的前提下继续存在；新页面不能继续以手写 `UiId` 或只返回 ID 的命中测试为基础。

迁移时先采用 `UiNode::auto()`、`UiKey` 和 `action_hit_at`。这不要求把 Grid 场景重写为 Pixel UI，也不要求现有编辑器立即迁移。

## 不写入现状的内容

通用焦点系统、滚动、文本自动换行、文本测量和可访问性语义树仍是未来工作。它们可在 `future/` 写目标和验收条件，但不能在本目录中描述为当前 API、默认行为或兼容性保证。

Web UI/runtime 也不属于当前迁移范围。现有 native Grid 与 Pixel UI 的模型、资源和提交边界保持独立于浏览器 API。
