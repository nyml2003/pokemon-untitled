# 错误模型

> 分类：现状；最后核对：2026-07-20。
> 依据：各层公开错误类型、`AGENT.md` 和 runtime 错误处理路径。

## 预期失败有类型

用户输入、项目格式、资产缺失、状态冲突和外部资源失败都属于预期失败。各层以 `Result`、诊断或明确枚举返回，例如 `MapError`、`TileSemanticsError`、`GameError`、`SceneViewError`、`FramePlanError`、`MapStorageError` 和 `GpuRuntimeError`。

调用者根据类型处理失败，而不是匹配错误字符串。诊断可以包含路径、规则位置或结构化码；底层模型不直接写标准错误或结束进程。

## 错误停留在所属边界

| 层 | 可返回的失败 | 不应返回的失败 |
| --- | --- | --- |
| foundation/domain | 模型构造、规则、范围、引用和数据校验。 | `io::Error`、窗口、GPU 或平台句柄错误。 |
| application | 命令不合法、用例状态不允许、领域错误传播。 | 路径、文件、终端和 Winit/WGPU 错误。 |
| presentation | 资产键、地图投影、UI 构建/布局和帧计划错误。 | 磁盘读取或 surface 获取错误。 |
| adapter | I/O、容器、终端、授权、WGPU/glyphon 等错误，并转换为项目错误。 | 进程退出策略。 |
| runtime | 启动、参数、日志、窗口关闭和退出状态。 | 可复用的业务规则。 |

例如，`MapProjectReader` 将容器和地图失败收束为 `MapStorageError`；`NativeTarget` 将 WGPU 与文本编码失败收束为 `NativeTargetError`；`game-host` 决定是否打印并退出事件循环。

## panic 与不变量

正常外部输入不能依赖 `panic!`、`expect` 或字符串匹配。当前代码中的 `expect` 只用于已经由构造、固定常量或本地控制流建立的条件，例如固定资产键、正 viewport 或已创建的文本 GPU 资源。新增预期失败路径应扩展错误类型和测试，而不是新增 panic。

运行时可把错误格式化给用户，但不会把格式化后的文本回传给 domain/application 作为状态输入。
