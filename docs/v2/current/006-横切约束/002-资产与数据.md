# 资产与数据

> 分类：现状；最后核对：2026-07-20。
> 依据：`game-assets`、`game-asset-plan`、`map-assets`、`game-fs-assets`、`game-data` 和 `game-data-import`。

## 资产以键和计划流动

presentation 用 `AssetKey`、`AssetRequest`、解码图像和图块目录表达“需要什么”，不把资源根目录或 `PathBuf` 放进视图模型。`game-asset-plan` 根据游戏状态和页面需求生成请求；`game-fs-assets` 再用 `AssetCatalog` 将请求映射到本地文件并返回 `AssetBytes`。

```text
状态或视图需求
  -> AssetKey / AssetRequest
  -> AssetCatalog
  -> game-fs-assets 读取字节
  -> game-assets 解码图像
  -> NativeAssets 构造 atlas
```

目录清单、文件路径和读取失败属于 adapter/runtime。资产键、尺寸检查和 atlas 计划属于 presentation/foundation，因此同一视图可以在不触及文件系统时测试。

## 地图资源

`map-assets` 根据图块资源构造地图资产描述；`map-render::AtomicTileCatalog` 将 `AtomicTileId` 映射为 `AssetKey`。`game-fs-assets::read_tile_sources` 从目录清单中筛选 `map/tile/` 前缀，并读取图块字节。`map-render` 只使用目录和地图模型投影图层，不读取磁盘。

`NativeAssets` 由解码后的 `(AssetKey, DecodedImage)` 列表创建。它分配资源 ID、建立 GPU atlas，并拒绝重复键或资源数超限；真正的 WGPU 资源创建仍在 `NativeTarget`/`GpuRuntime`。

## 游戏数据与导入

`game-data` 是只读领域数据。当前数据集和图鉴数据由 `include_bytes!` 嵌入编译产物，运行时不扫描 CSV 或访问网络。`game-data-import-core` 负责解析和校验输入数据；`game-data-import` 负责读取 PokeAPI CSV 目录、序列化 JSON、重新通过 `CurrentDataSet::from_json` 校验，并以临时文件替换目标。

导入工具是开发期 adapter。它产生的数据可被 `game-data` 内嵌，但不会成为产品游戏在运行时读取外部目录的依赖。
