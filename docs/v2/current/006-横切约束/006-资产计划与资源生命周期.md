# 资产计划与资源生命周期

> 分类：现状；最后核对：2026-07-20。
> 依据：`game-asset-plan`、`game-fs-assets`、`game-assets`、`game-native-plan`、`game-native-target` 与 `game-host`。

## 四段式资源链路

资源不从页面直接跳到文件系统。当前链路分为需求、读取、解码和 native 绑定四段：

```mermaid
flowchart LR
    demand[游戏与页面需求] --> request[AssetRequest\\nAssetKey]
    request --> catalog[AssetCatalog]
    catalog --> bytes[game-fs-assets\\nAssetBytes]
    bytes --> image[DecodedImage]
    image --> native[NativeAssets\\natlas 与 ResourceId]
    native --> plan[FramePlan]
    plan --> target[NativeTarget]
```

```text
游戏/页面状态
  -> AssetRequest（键、预期尺寸等）
  -> AssetCatalog + 本地字节
  -> DecodedImage
  -> NativeAssets（atlas + ResourceId）
  -> FramePlan 对 ResourceId 的引用
```

每段只验证自己能知道的事实。页面知道需要哪个 `AssetKey`，目录知道键对应哪个 source，解码器知道字节是否为可用图像，native assets 知道 atlas 是否能容纳资源。这样缺失资源能在计划阶段返回明确错误，而不是在 render pass 中以空白像素表现。

## 资产目录是权威映射

`AssetCatalog` 把稳定键映射为描述符和 source 路径。`game-fs-assets::load_catalog` 从 `catalog/assets.v1.json` 读取并校验目录；`read_asset_requests` 只读取目录中存在的键。调用者不能通过任意相对路径越过目录直接把文件带入游戏视图。

地图图块遵循相同规则：`read_tile_sources` 只读取 `map/tile/` 前缀的目录项，`map-assets`/`map-render` 将图块 ID 连接到资产键。地图模型只引用图块 ID，不保存磁盘路径。

## Atlas 的构造和稳定性

`NativeAssets::new` 按输入列表顺序为每个 `AssetKey` 分配从 1 开始的 `ResourceId`，拒绝重复键和超出 `u32` 的资源数，然后用 `game-assets::build_atlas` 构造 atlas。`ResourceId` 只在这一份 `NativeAssets` 内有意义；不能序列化进地图、游戏快照或 UI 状态。

这意味着资源重载必须重新构造一组一致的 `NativeAssets` 与后续 `FramePlan`。旧计划不能和新 atlas 混用，页面也不应缓存 resource ID。

## 生命周期与失败位置

| 阶段 | 典型失败 | 所在层 |
| --- | --- | --- |
| 请求规划 | 固定键、请求组合或预期尺寸不成立。 | presentation。 |
| 本地读取 | 目录缺失、键不存在、文件不存在或无法读取。 | `game-fs-assets` adapter。 |
| 解码/atlas | 图像无效、尺寸不符、重复键、atlas 构建失败。 | presentation。 |
| 帧计划 | 请求的资源不在 `NativeAssets`。 | `FramePlanError::UnknownAsset`。 |
| 提交 | native GPU 资源、surface 或 command 编码失败。 | adapter/runtime。 |

新增资产类型时，应先定义稳定键、目录项、请求和解码规则，再决定页面如何使用它。不要让新页面直接打开文件，也不要让 domain/application 持有图像、atlas 或 `PathBuf`。
