# 纯核心边界与可测试性

> 分类：现状；最后核对：2026-07-20。
> 依据：`game-session`、`world-application`、`map-editor-core`、`map-project-storage`、`game-host` 与各 crate 测试。

## 关键决策

业务状态转换以“取得旧值，返回新值和结果”的形式暴露。外壳负责取得外部输入和执行副作用。这个选择让同一规则可被窗口、CLI、测试和未来 runtime 重用。

| 边界 | 输入 | 输出 | 明确不拥有的东西 |
| --- | --- | --- | --- |
| `GameSession` | `GameCommand` | 新 session + `GameEvents` 或 `GameError` | 时钟、窗口、文件、GPU。 |
| `WorldApplication` | 世界命令或一次 NPC 推进 | 新 application + 观察/错误 | 地图路径、脚本文件、事件循环。 |
| `EditorModel` | `EditorIntent` | 新 model + `EditorEffect` 或 `MapError` | 鼠标设备、文件、标准输入输出。 |
| `MapProjectReader/Writer` | 字节或 `Read`/`Write` | 地图、字节或 `MapStorageError` | 路径选择、目录扫描、覆盖策略。 |

这不是把所有状态都复制一份。拥有权转移使旧状态在转换后不可再被使用；`EditorModel::reduce(&self)` 则显式 clone 一个下一状态，方便 UI 和 CLI 共享 reducer。两种接口都把状态更新收口在核心对象中。

## 外壳只能翻译

runtime/adapter 可做三件事：把平台事件翻译为核心输入、把核心输出投影给用户、执行核心请求的外部 effect。它们不得重新解释业务状态机。

例如 `EditorIntent::Save` 的 reducer 只返回 `EditorEffect::SaveRequested`。`map-editor` 与 `map-editor-cli` 分别选择路径和格式后写文件。若把 JSON 或 G3MP 写入放进 `EditorModel`，编辑器单元测试就需要临时目录，CLI 也会形成第二套保存规则。

同样，`game-host` 将 Winit 事件转为 `KeyEvent` 和 `TextEvent`，再通过表现动作形成 `GameCommand`。`GameSession` 不知道按键、修饰键、窗口焦点或真实时间。

## 不变量与测试位置

核心测试覆盖不变量，外壳测试覆盖翻译和环境：

| 不变量 | 证据位置 | 外壳不能替代的原因 |
| --- | --- | --- |
| 错场景不能提交世界/战斗命令。 | `game-session` 的 `WrongScene` 和测试。 | UI 是否禁用按钮不是状态保护。 |
| 未结束战斗不能返回世界。 | `GameError::BattleNotFinished`。 | runtime 不应猜测对战生命周期。 |
| 被使用的组合素材不能删除。 | `EditorModel::delete_material` 与单元测试。 | 视图隐藏删除按钮仍可被 CLI 绕过。 |
| 容器长度、校验和和地图引用必须一致。 | `MapProjectReader` 与 storage 测试。 | 文件存在不代表内容可信。 |

测试 fixture 应传入内存地图、资产 ID、脚本和命令，不依赖真实窗口或目录。需要真实文件、Winit 或 WGPU 时，才移动到 adapter/runtime 验证。

## 反模式

- 在窗口回调中直接修改 `MapProject` 或 `World` 字段。
- 在 CLI 子命令中复制 `EditorModel` 的撤销、素材删除或语义判断。
- 在 domain/application 保存 `PathBuf`、Winit 类型、surface 或 GPU device。
- 将 `io::Error`、WGPU 错误码或格式化文本当作领域状态。

这些做法会让入口之间的行为漂移，并使正常失败无法在无环境测试中复现。
