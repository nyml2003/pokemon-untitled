# 构建块与依赖

> 分类：现状；最后核对：2026-07-20
> 依据：根 `Cargo.toml`、各 crate 的 `Cargo.toml` 和根模块导出面。

## 分层

| 层 | 数量 | 职责 | 不能承担的职责 |
| --- | ---: | --- | --- |
| `foundation` | 6 | 网格、输入、GPU 计划、终端、UI 和 Ramus 的通用模型。 | 游戏规则、文件、窗口和平台 API。 |
| `domain` | 9 | 战斗、游戏数据、地图、瓦片语义、世界和叙事的规则与模型。 | 文件读写、窗口、GPU、网络和真实时间。 |
| `application` | 6 | 用例编排、会话和编辑器状态转换。 | 平台事件、文件加载和渲染提交。 |
| `presentation` | 10 | 资产键、视图、UI、场景投影、地图渲染和帧计划。 | 文件系统、Winit、WGPU 和状态规则变更。 |
| `adapter` | 8 | 文件、存储、导入、终端、Ramus 和 WGPU 集成。 | 领域规则和产品状态转换。 |
| `runtime` | 4 | 可执行程序装配、路径、平台事件、错误展示和退出。 | 新增游戏规则。 |
| `quality` | 2 | 端到端和环境验证。 | 产品运行时依赖。 |

依赖方向固定为：

```text
runtime -> adapter -> presentation -> application -> domain -> foundation
```

依赖规则限制方向，不要求每个 crate 经过所有中间层。例如，`punctum-wgpu` 可以直接使用 `punctum-gpu` 的提交模型。

## Foundation

| crate | 职责 |
| --- | --- |
| `punctum-grid` | 离散几何、稠密表面和有界帧 patch。 |
| `punctum-input` | 平台无关的键盘与已提交文本输入合同。 |
| `punctum-gpu` | GPU atlas、viewport、提交计划与字节编码。 |
| `punctum-terminal` | 终端 cell、文字、尺寸和 patch 计划。 |
| `punctum-ui` | 与渲染器无关的像素 UI tree、布局、绘制和命中模型。 |
| `ramus-core` | 类型化 capability tree、授权、解析、编译和执行前检查。 |

## Domain 与 Application

| 能力 | 领域模型 | 用例或状态所有者 |
| --- | --- | --- |
| 对战 | `battle-domain` | `battle-application`、`battle-session`、`game-session` |
| 世界 | `world-domain`、`world-project` | `world-application`、`game-session` |
| 地图 | `map-project`、`map-tile-semantics` | `map-editor-core`、`tile-editor-core` |
| 叙事 | `narrative-token`、`narrative-cps`、`narrative-compiler` | `world-application` |
| 游戏数据 | `game-data` | `game-session` 和视图投影读取 |

`game-session` 是单局产品会话。它协调世界与战斗场景。`world-application` 管理世界规则之外的外观与 NPC 脚本进度。`map-editor-core` 和 `tile-editor-core` 分别拥有地图编辑与瓦片语义编辑的状态转换。

## Presentation

| 组 | crate | 责任 |
| --- | --- | --- |
| 资产 | `game-assets`、`map-assets`、`game-asset-plan` | 资源键、解码图像、地图资产装配和加载计划。 |
| 游戏界面 | `game-ui`、`game-ui-kit`、`game-view` | 表现状态、可复用像素 UI 组件和游戏视图投影。 |
| 地图与场景 | `map-render`、`map-editor-view`、`game-scene-view` | 地图、编辑器和完整游戏场景投影。 |
| 提交计划 | `game-native-plan` | 将 Grid view 或 UI frame 变成 `FramePlan`。 |

presentation 只读取快照和资源目录。它生成 `GameView`、`UiTree`、`UiFrame`、`SceneFrame` 或 `FramePlan`，不提交 GPU 命令。

## Adapter、Runtime 与 Quality

| 组 | crate | 责任 |
| --- | --- | --- |
| 资产与存储 adapter | `game-fs-assets`、`map-project-storage` | 本地资产读取；版本化地图容器的编解码。 |
| 工具与能力 adapter | `game-data-import-core`、`game-data-import`、`battle-ramus-adapter`、`punctum-crossterm` | 数据导入、Ramus 对战能力和终端集成。 |
| 图形 adapter | `punctum-wgpu`、`game-native-target` | Winit 输入、WGPU runtime 和 `FramePlan` 的 native 提交。 |
| runtime | `game-host`、`map-editor`、`map-editor-cli`、`tile-editor` | 可执行程序的装配。 |
| quality | `game-e2e`、`wslg-wgpu-clear-smoke` | 游戏流程和 WSLg/WGPU 环境验证。 |

## 跨层数据合同

```text
输入：平台事件 -> punctum-input 类型 -> PresentationState / Command
状态：Command -> application/domain 状态转换 -> Event + Snapshot
视图：Snapshot -> GameView / UiFrame / SceneFrame -> FramePlan
输出：FramePlan -> adapter -> 窗口与 GPU
```

命令、事件、快照和帧计划是跨层传递的数据。调用方不能通过可变引用绕过这些状态转换入口。
