# 依赖规则与数据合同

> 分类：现状；最后核对：2026-07-20。
> 依据：根 `Cargo.toml`、各 crate manifest、`AGENT.md` 与当前公开类型。

## 允许的依赖

依赖从外向内：

```text
runtime -> adapter -> presentation -> application -> domain -> foundation
```

这是一条允许方向，不要求每个 crate 经过所有层。例如 `punctum-wgpu` 可以直接使用 `punctum-gpu` 的 `SubmissionPlan`，而 `game-scene-view` 可以读取 `game-session` 的快照。禁止的是内层为取得便利而反向依赖 adapter/runtime，或让 presentation 持有外部对象。

以下依赖需要拒绝或拆分：

- domain/application/presentation 引入 Winit、WGPU、`std::fs`、平台句柄或运行时路径。
- UI 或 renderer 通过可变引用直接改写 `GameSession`、世界或编辑器模型。
- runtime 为了处理一个用例而复制领域校验或状态机。
- adapter 将原始库类型泄漏为 domain/application 的错误或状态字段。

## 跨层合同

| 阶段 | 输入 | 输出 | 边界 |
| --- | --- | --- | --- |
| 输入 | 平台事件 | `KeyEvent`、`TextEvent`、编辑意图或 `GameCommand` | adapter/runtime 归一化。 |
| 状态转换 | 命令 | 更新后的 application 状态、事件或结构化错误 | application/domain。 |
| 观察 | application 状态 | `GameSnapshot`、`WorldObservation` 等只读数据 | application。 |
| 投影 | 快照、表现状态、资源描述 | `GameView`、`UiFrame`、`SceneFrame` | presentation。 |
| 提交计划 | 已解析视图和 `NativeAssets` | `FramePlan` | presentation。 |
| 外部输出 | `FramePlan` | `PresentOutcome` 或提交错误 | adapter/runtime。 |

这些合同使状态转换、场景投影和 GPU 提交可以分别测试。调用方通过命令和快照跨越状态边界，不通过内部字段或跨 crate 可变引用绕过它。

## 新能力的归属判断

| 问题 | 应放位置 |
| --- | --- |
| 它是否定义可复用的无业务模型或编码？ | foundation。 |
| 它是否定义游戏事实、规则、ID 或校验？ | domain。 |
| 它是否编排规则并拥有用例状态？ | application。 |
| 它是否只决定展示内容或生成提交计划？ | presentation。 |
| 它是否读取外部系统、调用库或转换平台事件？ | adapter。 |
| 它是否只选择具体实现、路径和进程生命周期？ | runtime。 |

归属仍不明确时，先定义输入和输出的数据合同。若合同包含文件路径、Winit/WGPU 类型、设备句柄或真实时钟，外部边界尚未被隔离；应先收窄到 adapter/runtime。
