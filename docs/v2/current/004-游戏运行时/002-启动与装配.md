# 启动与装配

> 分类：现状；最后核对：2026-07-20。
> 依据：`crates/runtime/game-host/src/main.rs`、`map.rs`、`narrative.rs`、`sprites.rs`。

## 启动顺序

`CreatureGameApp::new` 先构造不依赖窗口的游戏状态和资产，再进入 Winit 事件循环。顺序如下：

```text
读取地图项目和瓦片图像
  -> 编译期嵌入的叙事脚本
  -> WorldApplication::from_map_project_with_scripts
  -> CurrentDataSet::embedded 与 GameSession::new
  -> sprite manifest、图鉴和 NativeAssets
  -> 保存 runtime 所需的地图、目录、表现状态与时钟
```

地图默认来自 workspace 的 `maps/verdant-route/world.json`，资产目录从 workspace 根 `assets/` 解析。demo 叙事脚本用 `include_str!` 编译进 `game-host`。`GameSession::new` 接收已构造的 `WorldApplication`、内嵌数据和 roster seed；它不需要知道路径或窗口。

## 窗口与 native target

Winit 的 `resumed` 生命周期阶段调用 `initialize`。该方法创建标题为“宝可梦：还没想好名字”的窗口，初始逻辑尺寸为 960x720，读取实际像素尺寸，并构造 `NativeTarget`。`NativeTarget::new` 创建 WGPU instance、`GpuRuntime` 与 native 文本渲染器。

窗口、`NativeTarget`、资产、地图目录、输入修饰键和重绘计划都由 `CreatureGameApp` 持有。`GameSession` 以 `Option` 暂存，只为把拥有权移交给 `transition` 或 `advance_world_tick`，随后由 runtime 放回；窗口或 GPU 对象不会进入 session。

## 启动失败的归属

加载地图、编译脚本、构造世界、创建 session、读取图像、创建窗口和创建 GPU target 都可能失败。`game-host` 将这些错误映射到进程启动结果；它不把 `std::io::Error`、Winit 或 WGPU 类型传入 application/domain。

启动成功后，runtime 请求第一次重绘。游戏状态、投影和提交从此在事件循环中推进，而不是在构造函数中执行。
