# 帧计划与提交协议

> 分类：现状；最后核对：2026-07-20。
> 依据：`game-native-plan`、`game-native-target`、`punctum-wgpu`、`game-scene-view` 与 `game-host::redraw`。

## 关键决策

`FramePlan` 是 presentation 和 native adapter 之间的提交协议。它不包含 WGPU device、queue、surface、window 或文件路径；它只包含一组 `FramePass`。每个 pass 由 GPU `SubmissionPlan`、`NativeTextLabel`、viewport 和 `TextScale` 组成。

这使得“画什么”和“怎样取得 surface、编码 command buffer、present”可独立演进。`game-host` 可以在调用 native target 前完整构造和验证计划。

## 从场景到 pass

`SceneFrame` 分为 Grid、UI 和带 overlay 的组合形式。转换规则是：

| `SceneFrame` | 计划构造 | pass 数 |
| --- | --- | ---: |
| `Grid` | `FramePlan::from_game_view` | 1 |
| `Ui`、`Pokedex` | `FramePlan::from_ui_frame` | 1 |
| `GridWithUi`、`UiWithUi`、`PokedexWithUi` | 分别构造 base 与 overlay 后 `FramePlan::compose` | 2 |

`compose` 只组合已经独立有效的计划。overlay 不接触 base 的内部 GPU 数据；提交顺序由 pass 顺序表达。这是控制台覆盖世界、图鉴或战斗 UI 的方式。

## 资源解析的时机

`NativeAssets` 在帧计划之前从 `(AssetKey, DecodedImage)` 创建 atlas 和稳定 `ResourceId`。`from_game_view` 和 `from_ui_frame` 在计划阶段检查资源键是否存在，缺失时返回 `FramePlanError::UnknownAsset`，而不是在提交时发现一个空纹理。

UI path 还要求 `solid/white` 作为填充和边框基础资源；Grid path 要求各 layer 的 surface 尺寸一致。计划阶段可报告 `MissingSurface`、`SurfaceSizeMismatch`、无效 UI 内容和底层 GPU 计划错误。这些失败尚未触及 WGPU。

## 文本是覆盖层

`FramePlan` 不把文字烘焙为 atlas 图像。它保存 `NativeTextLabel`，包括逻辑格坐标、跨格宽高、内容、颜色和可选字号。`NativeTarget::present` 先将每个 pass 的 GPU plan 交给 `GpuRuntime::present_plans_with_overlays`，再在对应 pass 的 encoder 中用 glyphon 编码文本。

文本 renderer 以 surface format 为缓存键；format 变化时重新创建文本 GPU 资源。这样 presentation 只声明文本，native target 决定字体系统、shape、atlas、viewport 和 WGPU render pass。

## Surface 获取策略

`GpuRuntime` 将 surface 结果翻译为 `PresentOutcome`：成功、成功后重配、最小化跳过、超时跳过、遮挡跳过、仅重配或丢失。非正常但可恢复的 outcome 不变成 `FramePlan` 错误，也不回写游戏状态。

`game-host` 仅对 `Reconfigured` 与 `SurfaceLost` resize 后请求重绘。超时、遮挡和最小化跳过当前帧，但不撤销已经发生的游戏命令。不可恢复的 `NativeTargetError` 才由 runtime 输出并退出。

## 维护约束

新增渲染路径时，先决定它是 `GameView` 还是 `UiFrame`，再定义能否与 overlay 组合、所需资源和文本语义。不要让 presentation 直接持有 WGPU 类型，也不要让 native target 按页面名称猜测绘制规则。
