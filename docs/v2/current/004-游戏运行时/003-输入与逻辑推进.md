# 输入与逻辑推进

> 分类：现状；最后核对：2026-07-20。
> 依据：`game-host/src/main.rs`、`punctum-wgpu/src/input.rs`、`game-ui` 与 `game-session` 的公开 API。

## 输入先归一化

`game-host` 从 Winit 接收键盘、IME 文字、鼠标和窗口事件。键盘事件先包装为 `WinitKeyEventSnapshot`，再调用 `normalize_key_event`；已提交文字通过 `WinitCommittedTextSnapshot` 和 `normalize_committed_text` 转换。无效文字输入只在 runtime 输出诊断，不进入游戏状态。

这样 `game-ui` 与 `game-session` 只看到 `punctum-input` 的平台无关类型，而不是 Winit 的原始类型。鼠标位置、按钮和修饰键仍由 runtime 保存，用于当前 UI 命中与窗口交互。

## 表现动作与游戏命令

按键到状态变化分两步：

```text
KeyEvent + TextEvent + GameSnapshot
  -> PresentationState::handle_key
  -> PresentationUpdate
  -> runtime 处理 PresentationAction
  -> GameCommand（仅需要改变游戏时）
  -> GameSession::transition
```

`PresentationState` 管理控制台、菜单、图鉴、焦点与动画。它可消耗输入而不产生游戏命令，例如打开 UI 或移动表现焦点。需要改变世界或战斗时，runtime 将 action 转为 `GameCommand`，调用 `submit_game`，并保存返回的新 session。UI 从不直接修改 `GameSession` 字段。

## 真实时间与逻辑推进

`CreatureGameApp` 持有 `Instant`、下一次 world tick 和下次唤醒时间。当前 `WORLD_LOGIC_TICK` 为一秒。只在 `GameScene::World` 活动时，runtime 才调用 `GameSession::advance_world_tick`；战斗场景不以这个时钟推进世界。

world tick 可能推进 NPC 脚本和自主世界行为，但它接收的是一次离散调用，不读取系统时钟。真实时间、动画节奏、重绘和事件循环唤醒都留在 runtime/表现边界，因此同一 application 状态转换可在测试中重复执行。
