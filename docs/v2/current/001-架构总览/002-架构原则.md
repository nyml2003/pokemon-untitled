# 架构原则

> 分类：现状；最后核对：2026-07-20。
> 依据：根 `Cargo.toml`、`AGENT.md`、各层 crate 的 `Cargo.toml` 与公开根 API。

## 结论

分层不是按目录归类，而是按可替换边界划分。游戏规则和用例不读取路径、不创建窗口、不提交 GPU 命令；运行时和 adapter 才把这些外部能力接到核心。

```text
runtime -> adapter -> presentation -> application -> domain -> foundation
```

箭头表示允许的依赖方向。内层不知道外层的 crate、平台库或执行入口。根 `Cargo.toml` 的 workspace 成员按这六层和 `quality` 组织，`AGENT.md` 将这条方向作为项目约束。

## 纯核心与外部能力

| 层 | 解决的问题 | 不应承担的能力 |
| --- | --- | --- |
| foundation | 网格、输入、GPU 计划、UI 基础模型和 Ramus 通用能力。 | 游戏规则、文件、窗口和平台调用。 |
| domain | 战斗、地图、世界、叙事和游戏数据的模型与规则。 | 文件系统、网络、窗口、GPU。 |
| application | 命令、状态转换和跨领域用例编排。 | 真实时间、窗口事件、资产目录。 |
| presentation | 从快照生成场景、UI 和提交计划。 | Winit、WGPU、文件读取和进程控制。 |
| adapter | 文件、二进制存储、终端、Winit/WGPU 与受控 Ramus 调用。 | 新增游戏规则。 |
| runtime | 创建具体实现，装配依赖并处理进程生命周期。 | 领域状态机和规则分支。 |

`game-fs-assets` 以 `Path` 读取目录并返回资产字节。`map-project-storage` 以字节流读写已校验地图。`game-native-target` 将 `FramePlan` 提交给 `punctum-wgpu`。这些实现都在 adapter，避免把 `std::fs`、Winit 和 WGPU 扩散到核心层。

## 状态转换与投影

游戏的权威状态在 application/domain 一侧。`GameSession::transition` 接受 `GameCommand`，返回更新后的 session 和事件结果；运行时不通过 UI 或渲染对象直接改写该状态。`GameSession::snapshot` 提供只读快照给表现层。

表现链路保持单向：

```text
归一化输入
  -> PresentationState / GameCommand
  -> GameSession::transition
  -> GameSnapshot
  -> project_scene
  -> SceneFrame
  -> FramePlan
  -> NativeTarget::present
```

`game-host` 的 `redraw` 按此顺序取得快照、调用 `project_scene`、构造 `FramePlan` 并调用 `NativeTarget::present`。`game-scene-view` 和 `game-native-plan` 位于 presentation；`NativeTarget` 位于 adapter。这样场景投影和帧计划可在没有窗口、GPU 或真实资产路径时构造和测试。

## 失败边界

预期失败以 `Result` 和结构化错误表达。地图二进制读取返回 `MapStorageError`，其中包含截断、版本、校验和、尺寸上限和领域校验失败；WGPU 提交返回 `NativeTargetError` 或 `PresentOutcome`。平台或库错误在 adapter/runtime 解释，不成为 domain/application 的平台错误码。

运行时决定进程级后果。例如 `game-host` 在场景投影、帧计划或提交失败时输出错误并退出事件循环；表面丢失或需要重配时依据 `PresentOutcome` 调整目标并请求重绘。渲染结果不反向修改游戏规则。
