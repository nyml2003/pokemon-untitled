# 证据与维护

> 分类：现状；最后核对：2026-07-20。
> 依据：`docs/v2/CONVENTIONS.md`、根 `Cargo.toml`、crate 源码、测试和 `AGENT.md`。

## 证据顺序

现状页面只能写已经存在的结构和行为。出现冲突时，以下顺序决定结论：

1. 当前源码和公开类型决定实际行为与边界。
2. `Cargo.toml` 决定 workspace 成员、package 类型和声明依赖。
3. 测试决定已被自动验证的规则与失败路径。
4. `AGENT.md` 决定当前开发、分层和平台约束。
5. 命令实际输出决定某次环境验收是否通过。

文档不是更高优先级的事实来源。代码变化使页面陈述失真时，应当在同一变更中改正文档，或删除暂时无法证明的句子。

## 页面与变更的对应关系

| 发生变化 | 必须复核的页面 |

topic `README.md` 只维护本目录的阅读顺序和页面边界。具体事实写入编号内容页，避免导航页变成第二份架构说明。

## 现状与未来的分离

`current/` 写现在可运行、可读取或可验证的事实。页面开头必须标明分类和最后核对日期。`future/` 写尚未实现的目标、前置条件和验收标准；它不证明当前行为。

方案被替代或完成时，不保留过时目录、跳转页或兼容性说明。旧版本由 Git 历史保存。此规则同样适用于本文档体系的目录与文件名。

## 复核方式

修改现状说明后至少检查：

- 相对链接仍能定位到目标页面。
- 页面的代码路径、crate 名和命令名仍存在。
- 依赖、状态流或错误处理的陈述能定位到具体源码。
- Markdown 没有空白错误，使用 `git diff --check -- docs` 检查。

涉及实际行为时，验证命令必须遵循 [开发与验收](../008-开发与验收/README.md) 中的 `ops` 工作流；不要在文档中把未执行的命令写成已通过的验收。
