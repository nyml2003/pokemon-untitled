# 游戏基座与薄切片验证

> 分类：未来；最后整理：2026-07-20。
> 状态：提案。本文不代表功能已经存在，也不承诺实现排期。

## 结论

先建设可持续游戏所需的状态和内容合同。不要先建设背包菜单、商店画面或完整任务 UI。

每个基座完成后，必须由一个很薄的上层流程验证。这个流程可以是纯 application 测试或最小 host，不要求先完成正式表现层。

目标不是尽快制作大量地图和剧情。目标是让同一份游戏状态能可靠地支持地图、NPC、战斗、物品、金钱和存档。

## 当前依据

以下能力已经存在：

- `world-domain` 持有网格、角色位置、朝向、阻挡和草丛遭遇规则。
- `WorldApplication` 将 `MapProject` 映射为可运行世界，并保存外观、NPC 脚本进度和 NPC 对白。
- `GameSession` 持有世界与对战场景，并在草丛遭遇时创建对战 session。
- 地图项目已经有碰撞、遭遇 cell 和角色；地图编辑器、JSON 与 G3MP 存储也已存在。
- narrative DSL 已能编译 actor 绑定的移动、朝向和对白脚本。

当前 NPC 脚本由 `advance_npcs` 按逻辑 tick 自动推进。它还没有玩家确认键触发、条件分支、持久化动态状态或通用副作用。当前对战队伍也由 roster seed 生成，尚未作为玩家持久队伍回写。

依据：`world-domain`、`world-application`、`game-session`、`map-project`、地图编辑器与 `narrative-*` crate；详见现状文档中的[状态所有权](../current/004-游戏运行时/001-状态所有权.md)和[地图到世界的映射](../current/005-地图创作/008-地图到世界的映射.md)。

## 状态边界

未来应引入一个纯游戏状态聚合。名称可以是 `GameState`，但名称不是重点。

```text
GameState
  PlayerState
    - 当前地图和位置
    - 队伍与每只精灵的可变战斗状态
    - 背包和金钱
    - 徽章、事件 flag 与变量
  WorldState
    - 各地图的动态对象状态
    - 已拾取物品
    - 已击败训练师
    - NPC 当前行为和脚本进度
  SessionState
    - 当前场景和可恢复的临时交互
    - 随机数状态或明确的随机输入记录
```

静态内容和动态存档必须分开：

| 类型 | 放置内容 | 不放置内容 |
| --- | --- | --- |
| 内容定义 | 地图布局、连接、NPC 外观、对话脚本、训练师队伍、商店货品、物品定义 | 玩家位置、已击败状态、已领取状态、当前金钱 |
| 游戏状态 | 地图位置、队伍 HP/PP/经验、物品数量、金钱、flag、动态对象变化 | 图块图片路径、重复的 NPC 静态数据、整份脚本 |

领域与 application 持有这份状态。presentation 只能读取快照；runtime 和 adapter 只负责输入、资产和存储边界。

## NPC 模型

NPC 不是 `Trainer` 的同义词。`Trainer` 是 NPC 可选能力之一。

每个 NPC 由通用角色、行为、交互和可选能力构成：

```text
NpcDefinition
  Actor：稳定 ID、初始位置、朝向、外观、碰撞
  Behavior：静止、巡逻、路线移动、注视、跟随、出现/消失
  Interaction：面对后可调用的事件入口
  Capabilities：Trainer、Merchant、Healer、Gift、Guide、Blocker 等
```

`Actor` 只表示空间实体。`Interaction` 决定玩家按确认键后的事件入口。`Behavior` 决定非玩家推进。能力只提供该 NPC 所需的额外数据和触发条件。

不要用不断增长的 `NpcKind` 枚举承载全部组合。一个角色可能巡逻、第一次送物品、之后改变对白并在条件满足后挑战玩家。上述组合应由状态、事件和能力表达。

动态状态只保存可变部分，例如“礼物是否已领取”“训练师是否已击败”“当前路线节点”“当前脚本 continuation”。静态定义始终由内容数据读取。

## 内容合同的建设顺序

### 1. 持久游戏状态和可复现命令

先定义状态、命令、事件、错误和快照。不直接写文件。

每条命令必须给出前置条件、成功事件、失败类型和状态变化。随机遭遇、捕获或战斗波动必须接受明确随机输入，或保存可恢复随机状态。

完成条件：相同初始状态和相同命令/随机输入总能得到相同结果；失败不会破坏原状态。

### 2. 世界地图和对象合同

地图从单张网格扩展为世界内容：稳定地图 ID、连接、warp、出生点、对象、触发区和独立遭遇表。

`Encounter` cell 可以继续表示“此处允许遭遇”，但不能单独承担精灵池、概率和等级规则。遭遇区域必须引用独立内容定义。

完成条件：编辑器和运行时都能校验连接目标、warp 目标、对象 ID、触发入口和遭遇表引用。

### 3. 事件运行时

先支持最小事件指令：对白、选择、flag/变量读写、给物品、加减金钱、warp、开始战斗、治疗和结束。

事件执行必须返回类型化结果。事件不能通过 presentation 或 runtime 直接修改游戏状态。

完成条件：条件分支、一次性拾取、重复对话和失败路径都可由纯测试验证。

### 4. NPC 的交互和行为

在通用 actor 基础上增加面对对象、确认键交互、行为推进和能力触发。训练师视线触发和战后状态是这个模型的一个消费者，不是模型本身。

完成条件：静止对话 NPC、一次性赠送 NPC、阻路 NPC 和训练师都不需要新增空间角色类型。

### 5. 玩家队伍和战斗结果回写

把玩家队伍从 demo roster 提升为持久状态。战斗结果返回经验、HP、PP、异常、金钱、训练师状态和事件状态的明确变更。

完成条件：对战开始读取当前队伍；结束后只回写合法结果；中断或非法命令不丢失原状态。

### 6. 物品和经济领域模型

先实现 `ItemId`、分类、数量上限、容量、价格和金钱边界。先由事件和测试使用它们。

背包菜单、商店页面、卖出、携带道具、制作和复杂经济规则后置。

完成条件：拾取、消费、购买、余额不足、背包已满和数量溢出都有明确结果。

### 7. 存档格式和迁移

只在游戏状态合同稳定后定义存档 envelope。它至少包含格式版本、内容兼容信息、游戏状态和完整性校验。

完成条件：新旧版本迁移、损坏输入、未知内容 ID 和重载后的状态一致性均有测试。

## 薄切片验证

不要等所有系统完成才验证。每完成一个基座，就扩展同一个小流程：

```text
新游戏
  -> 在起始镇与 NPC 交互，获得精灵和伤药
  -> 经 warp 进入路线
  -> 草丛遭遇并完成战斗
  -> HP、PP、经验和金钱回写
  -> 训练师挑战且只可完成一次
  -> 商店购买一个物品
  -> 存档，重载后得到相同状态
```

首选验证层级：

1. domain：值对象、不变量、命令失败和确定性规则。
2. application：从初始状态执行整段命令，断言事件和最终快照。
3. runtime：只验证输入归一化、装配和至少一条实际可操作路径。

不应以截图或手工试玩代替前两层测试。也不应为了测试而先实现完整菜单和美术。

## 不在本提案中决定的事项

- 具体地区、剧情、阵营、道馆和数值设计。
- 捕获、进化、特性、亲密度、双打和通信。
- 背包与商店的最终 UI。
- 存档媒介和平台实现，例如本地文件、浏览器存储或云同步。
- 地图编辑器的具体交互设计。

这些事项仍需各自的内容或技术提案。它们不得绕过本文定义的状态所有权和薄切片验证要求。

## 进入实现前的检查

- 目标状态属于静态内容、动态游戏状态还是表现瞬态状态。
- 命令、成功事件、失败类型和快照变化已明确。
- 地图/对象/脚本的稳定 ID 与引用校验已定义。
- 随机输入或随机状态的可复现策略已定义。
- 至少一项薄切片断言能证明此功能与已有基座共同工作。
