# 需要产品决策的问题

## 结论

以下问题不是代码细节。每个答案都会改变数据协议、状态所有权或发布边界。当前实现可以继续做小型体验改进，但在加入长期内容前，至少应先决定“项目做成什么产品”和“是否需要持久进度”。

## 1. 产品定位

| 选项 | 架构含义 | 适合当前情况 |
| --- | --- | --- |
| A. 可玩的 Gen3 风格单机切片 | 继续聚焦一张地图、一套演示队伍，优先完成手感和核心战斗 | 最低风险，可快速形成可试玩闭环 |
| B. 完整单机怪物 RPG | 需要玩家状态、存档、多地图、任务、NPC、遭遇配置 | 需要先冻结存档和地图事件模型 |
| C. 可编辑的通用 RPG 工具 | 地图、数据、脚本、发布管线必须成为产品 API | 投入最大，需要重新审视 Gen3 专有类型 |

当前默认假设是 A 向 B 演进。若目标实际是 C，应尽早停止把 demo 规则和资源键当成通用抽象。

## 2. 地图编辑器定位

| 问题 | 内部制作工具 | 对外产品 |
| --- | --- | --- |
| 文件格式 | 可以快速迭代，仍需最低迁移策略 | 必须版本化、文档化和兼容 |
| 错误处理 | stderr + 重新打开可接受 | 需要恢复、备份、冲突和用户提示 |
| 功能范围 | 先服务当前地图 | 实体、事件、资源浏览、项目管理、导出 |
| 测试重点 | 核心 reducer 与 JSON | 端到端交互、跨版本项目、发布包 |

## 3. Ramus 的边界

Ramus 当前是“玩家可执行合法战斗动作”的受限命令适配层：单条、无参数、256 字节，且能力由 adapter 固定。需要决定它未来是：

1. 只保留为战斗控制台和调试辅助。此时继续保持很小的调用表和严格权限。
2. 用作游戏事件/NPC 规则语言。此时先定义脚本包、版本、沙箱、权限、失败/重试和存档语义。
3. 用作通用 mod API。此时需要公开稳定的数据模型、兼容政策和安全威胁模型。

第二和第三项不是“多加几个 provider”即可完成。它们会改变游戏内容安全和数据迁移策略。

## 4. 存档语义

| 决策 | 可选方向 | 架构影响 |
| --- | --- | --- |
| 保存时机 | 仅非战斗 / 任意时刻 | 是否序列化 RNG、回放和待选动作 |
| 数据升级 | 严格拒绝 / 迁移 / 内嵌旧 bundle | `CurrentDataSet` 与 asset 版本记录 |
| 地图升级 | 坐标保持 / 锚点迁移 / 回到安全出生点 | world registry 与地图 ID 协议 |
| 文件策略 | 单文件 / 多槽位 / 云同步 | adapter 接口、冲突与原子性 |

## 5. 发布目标

当前已实际装配的是 Winit/WGPU 原生桌面。终端 adapter 存在但没有产品 runtime。请确定近期优先级：

- 只做原生桌面：减少未使用后端的维护期望，优先 UI、性能和发布包。
- 加终端调试运行器：验证 `GameView` 与输入合同的可移植性，但要定义终端 UI 取舍。
- 做 Web：需要先验证 WGPU/WASM、文件加载、键盘/IME、资产发布和存档方案。

## 建议的回答顺序

1. 在 A/B/C 中选择未来 6-12 个月的产品定位。
2. 决定是否在下一批功能前实现持久进度。
3. 决定地图编辑器是内部工具还是对外产品。
4. 确定 Ramus 是否超出战斗控制台。
5. 选择第二发布目标，或明确原生桌面是唯一目标。

这些决定一旦明确，就可以把 [扩展点](002-extension-points.md) 变成分阶段实施计划，并为存档、世界事件、脚本或平台适配创建具体的 OEP/ADR 文档。
